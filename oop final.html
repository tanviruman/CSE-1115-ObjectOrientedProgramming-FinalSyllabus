The Exam Question (Reconstructed)

CT-04 Problem Statement

A file students.txt contains student information. Each line has: name, ID, CGPA (comma-separated). Some lines may have missing values. Write a Java program that reads the file and generates:

  1. uga_candidate.txt — students with CGPA ≥ 3.70, sorted in descending CGPA order.
  2. probation.txt — students with CGPA < 2.00, sorted in ascending ID order.
  3. errors.txt — all lines with one or more missing values.

Key Observations for Exam Success

  • You need two separate Comparator classes — one for CGPA (descending) and one for ID (ascending). Know how to flip direction.
  • The split(",", -1) with -1 limit is critical for detecting trailing missing fields.
  • You need three separate FileWriter objects — one per output file. Confusing them will cost marks.
  • Always call .close() on every Scanner and FileWriter. Leaving streams open is a common mistake.
  • The class uses int id (not String), so you must use Integer.parseInt() when reading and compare with Integer.compare().

Full Marks Strategy

Follow this exact approach in the exam hall. Each step prevents a specific category of lost marks.

  1. 1

    Read the full question before writing a single line

    Underline every output requirement. Count how many output files are needed. Identify the sort criteria for each (ascending vs descending, which field). Misreading a sort direction is a very common, very costly mistake.

  2. 2

    Sketch your class structure first (30 seconds)

    Write the class names you'll need as a mental map: Student, CgpaComparator, IdComparator, Main. Knowing your structure before coding prevents tangled rewrites.

  3. 3

    Write the Student class first — get the fields right

    Use the correct types: String name, int id, double cgpa. Add a toString() that formats exactly as required. If the question shows a sample output format, match it exactly.

  4. 4

    Write Comparator classes — double check direction

    For descending order: return Double.compare(s2.cgpa, s1.cgpa) (s2 first). For ascending order: return Integer.compare(s1.id, s2.id) (s1 first). Use the built-in Double.compare() and Integer.compare() — never subtract doubles.

  5. 5

    Write the File I/O block with the correct split argument

    Always use split(",", -1) when you need to detect missing fields. Check parts.length != 3 AND parts[i].trim().isEmpty(). Both conditions are required for full missing-value detection.

  6. 6

    Sort, then write to output files

    Sort each list before writing. Declare FileWriters with distinct variable names (fw1, fw2, fw3 — or better, descriptive names). End every write loop and close every writer.

  7. 7

    Handle exceptions correctly

    The simplest and most accepted pattern in class exams: declare public static void main(String[] args) throws Exception. This handles all checked exceptions (IOException, FileNotFoundException) without try-catch boilerplate. Only use try-catch if the question explicitly asks for it.

  8. 8

    Trace through your code mentally with one sample line

    Trace: "Alice,1,3.85" — does it reach uga? Trace: "Bob,,2.5" — does it reach errors? This 60-second mental simulation catches most bugs before submission.

File I/O

Java File I/O is the backbone of the exam question. You must read from a file line by line, parse each line, and write results to multiple output files. Two approaches appear in the lecture slides.

Approach 1: Scanner + FileWriter (Exam-Primary)

This is the approach used in the CT-04 solution and in Main.java. It is the most straightforward for exam conditions.

Reading a file with Scanner
import java.io.*;
import java.util.*;

// READING — throws Exception handles all checked exceptions
Scanner sc = new Scanner(new File("students.txt"));

while (sc.hasNextLine()) {
    String line = sc.nextLine();          // read one full line
    String[] parts = line.split(",", -1); // split on comma, keep empties
    // process parts[0], parts[1], parts[2]...
}
sc.close(); // ALWAYS close
Writing a file with FileWriter
// WRITING
FileWriter fw = new FileWriter("output.txt");

for (Student s : studentList) {
    fw.write(s.name + "," + s.id + "," + s.cgpa + "\n");
}
fw.close(); // ALWAYS close — data may not flush without this

★ The split(",", -1) Rule — Most Important Detail

split(",") silently discards trailing empty strings. split(",", -1) keeps them. This distinction is the entire mechanism for detecting trailing missing fields:

  • "Alice,1,3.85" → both give ["Alice","1","3.85"] (length 3) ✓
  • "Alice,," → split(",") gives ["Alice"] (length 1) — WRONG
  • "Alice,," → split(",", -1) gives ["Alice","",""] (length 3) — CORRECT ✓

Without -1, you cannot detect missing CGPA or ID at the end of a line. Use it every time.

Approach 2: BufferedReader + BufferedWriter

This is the approach shown in the FileIO.pdf lecture slides. It is more efficient for large files and is important to know for theory questions. The key difference from Scanner is that readLine() returns null at end-of-file (not hasNextLine()).

BufferedReader / BufferedWriter pattern (from lecture slides)
import java.io.*;

try {
    BufferedReader reader = new BufferedReader(new FileReader("B.txt"));
    BufferedWriter writer = new BufferedWriter(new FileWriter("A.txt"));

    String s;
    while ((s = reader.readLine()) != null) { // null = end of file
        writer.write(s);
        writer.newLine(); // platform-independent newline
    }

    reader.close();  // IOException
    writer.close();  // IOException

} catch (IOException e) {
    e.printStackTrace();
}

Complete Exam Solution — CT-04

This is the full solution to the CT-04 question with inline explanations of every important decision.

CT-04 — Full Java Solution
import java.io.*;
import java.util.*;

// ── Student class ──────────────────────────────────────
class Student {
    String name;
    int id;       // int, not String — allows Integer.compare()
    double cgpa;

    Student(String name, int id, double cgpa) {
        this.name = name;
        this.id   = id;
        this.cgpa = cgpa;
    }

    public String toString() {
        return name + ", " + id + ", " + cgpa;
    }
}

// ── Comparator: highest CGPA first (descending) ─────────
class CgpaComparator implements Comparator<Student> {
    public int compare(Student s1, Student s2) {
        return Double.compare(s2.cgpa, s1.cgpa); // s2 first = descending
    }
}

// ── Comparator: smallest ID first (ascending) ───────────
class IdComparator implements Comparator<Student> {
    public int compare(Student s1, Student s2) {
        return Integer.compare(s1.id, s2.id); // s1 first = ascending
    }
}

// ── Main class ──────────────────────────────────────────
public class Main {
    public static void main(String[] args) throws Exception {

        ArrayList<Student> uga       = new ArrayList<>();
        ArrayList<Student> probation = new ArrayList<>();
        ArrayList<String>  errors    = new ArrayList<>();

        Scanner sc = new Scanner(new File("students.txt"));

        while (sc.hasNextLine()) {
            String line   = sc.nextLine();
            String[] parts = line.split(",", -1); // -1 keeps trailing empties

            // Check: wrong number of fields OR any field is blank
            if (parts.length != 3           ||
                parts[0].trim().isEmpty() ||
                parts[1].trim().isEmpty() ||
                parts[2].trim().isEmpty()) {
                errors.add(line);
                continue; // skip to next line
            }

            String name  = parts[0].trim();
            int    id    = Integer.parseInt(parts[1].trim());
            double cgpa  = Double.parseDouble(parts[2].trim());

            Student s = new Student(name, id, cgpa);

            if      (cgpa >= 3.70) uga.add(s);
            else if (cgpa <  2.00) probation.add(s);
            // students between 2.00 and 3.70 are simply not written anywhere
        }
        sc.close();

        // Sort, then write each list
        Collections.sort(uga, new CgpaComparator());
        Collections.sort(probation, new IdComparator());

        FileWriter fw1 = new FileWriter("uga_candidate.txt");
        for (Student s : uga)       fw1.write(s + "\n");
        fw1.close();

        FileWriter fw2 = new FileWriter("probation.txt");
        for (Student s : probation) fw2.write(s + "\n");
        fw2.close();

        FileWriter fw3 = new FileWriter("errors.txt");
        for (String e : errors)     fw3.write(e + "\n");
        fw3.close();
    }
}

⚠ Common Pitfalls — File I/O

  • Using split(",") without -1: Trailing empty fields are silently dropped, so missing CGPA at the end of a line like "Alice,1," will not be detected. Always use split(",", -1) when validating missing values.
  • Forgetting .trim() before isEmpty(): A field that is just spaces (" ") would pass isEmpty() but is still invalid data. Always .trim() before checking.
  • Forgetting .close() on FileWriter: Data is held in an internal buffer. Without close(), the last few lines may never reach the file. This is a marks-killer that is hard to debug.
  • Mixing up FileReader vs File: Scanner takes a new File("name.txt") argument. BufferedReader takes a new FileReader("name.txt"). Using the wrong one causes a compile error.
  • Writing newlines incorrectly: fw.write(s.toString()) — if you forget the + "\n", all output will be on one line. With BufferedWriter, use writer.newLine() instead of "\n".
  • Not handling FileNotFoundException: new Scanner(new File(...)) throws FileNotFoundException. The cleanest solution in an exam is throws Exception on main.
  • Reading with next() instead of nextLine(): next() splits on any whitespace, not newlines. Always use nextLine() for line-by-line file reading.

Comparable

Concept & The compareTo Contract

The Comparable<T> interface defines the natural ordering of a class. A class that implements it can be sorted by Arrays.sort() or Collections.sort() without providing any external comparator.

When you implement Comparable, you override exactly one method:

public int compareTo(T other)

The return value has a precise meaning that you must memorize:

Return ValueMeaningResult in sorted order
Negative (e.g. -1)this < otherthis comes BEFORE other
Zero (0)this == otherorder unchanged
Positive (e.g. +1)this > otherthis comes AFTER other

ℹ Mental Model for direction

Think: "I want this value to go FIRST → return -1." If you want higher CGPA first (descending), and this.cgpa is higher, you want it first, so return -1.

Implementation — Multi-Level Sorting

The lecture code in Comparable.java demonstrates multi-level sorting: primary sort by CGPA (descending), secondary sort by ID (ascending) when CGPAs are equal. This is a common exam pattern.

Comparable.java — complete implementation with explanation
import java.util.Arrays;

class Student implements Comparable<Student> {
    String name;
    String id;
    double cgpa;

    Student(String name, String id, double cgpa) { ... }

    public int compareTo(Student other) {
        // PRIMARY: sort by CGPA descending
        if (this.cgpa > other.cgpa) return -1; // this has higher CGPA → comes first
        if (this.cgpa < other.cgpa) return  1; // this has lower CGPA → comes after

        // SECONDARY (CGPAs are equal): sort by ID ascending
        if (Integer.parseInt(this.id) < Integer.parseInt(other.id)) return -1;
        if (Integer.parseInt(this.id) > Integer.parseInt(other.id)) return  1;
        return 0; // completely equal
    }
}

public class Tst {
    public static void main(String[] args) {
        Student s1 = new Student("Adrita",  "001", 2.9);
        Student s2 = new Student("Badhon",  "002", 3.8);
        Student s3 = new Student("Chamely", "003", 3.8);
        Student[] array = {s1, s3, s2};

        Arrays.sort(array); // no comparator needed — uses compareTo()

        for (Student s : array) System.out.println(s);
    }
}

✓ Tracing the expected output

  • Badhon (3.8, id 002) and Chamely (3.8, id 003) tie on CGPA → secondary sort by ID → Badhon (002) comes first.
  • Adrita (2.9) has the lowest CGPA → comes last.
  • Output: Badhon → Chamely → Adrita

⚠ Common Pitfalls — Comparable

  • Forgetting the generic type: Writing implements Comparable instead of implements Comparable<Student> makes the parameter type Object, forcing a cast and losing type safety.
  • Flipping the sign for descending order: For descending, return -1 when this > other (you want the bigger value first). Students often get this backwards.
  • Subtracting doubles: Never do return (int)(this.cgpa - other.cgpa). Precision loss from casting can make 3.80 - 3.75 → 0. Always use explicit if-else comparisons or Double.compare().
  • Missing the zero return: If you forget return 0 at the end, the method won't compile (not all paths return a value). Always end with return 0.
  • Calling Arrays.sort() on a class without Comparable: Throws ClassCastException at runtime. If the class doesn't implement Comparable, you must provide a Comparator.
  • Parsing ID inside compareTo repeatedly: If id is stored as a String, calling Integer.parseInt() inside compareTo works but is inefficient. In the exam solution, id is int — no parsing needed.

Comparator

The Comparator<T> interface defines an external, reusable comparison strategy. Unlike Comparable, the comparison logic lives outside the Student class, and you can create as many Comparators as you need — one for each sort criterion.

Concept & Contrast with Comparable

A Comparator takes two arguments and decides their order. The sign logic is the same as compareTo: negative means s1 comes first, positive means s2 comes first, zero means equal.

class MyCmp implements Comparator<Student> {
    public int compare(Student s1, Student s2) {
        // return negative → s1 comes first
        // return positive → s2 comes first
        // return 0        → equal
    }
}

Multiple Comparators — From the Lecture Files

The lecture code provides two patterns. Study both — the CT-04 solution uses the cleaner Double.compare() / Integer.compare() style.

Pattern A — Manual if-else (from Comparator.java)
// Sort by CGPA — DESCENDING (highest first)
class CgBasedComparator implements Comparator<Student> {
    public int compare(Student o1, Student o2) {
        if      (o1.cgpa > o2.cgpa) return -1; // o1 bigger → o1 first
        else if (o1.cgpa < o2.cgpa) return  1;
        else                         return  0;
    }
}

// Sort by ID — DESCENDING (highest ID first) — note: opposite of CT solution
class IDBasedComparator implements Comparator<Student> {
    public int compare(Student o1, Student o2) {
        if      (Integer.parseInt(o1.id) > Integer.parseInt(o2.id)) return -1;
        else if (Integer.parseInt(o1.id) < Integer.parseInt(o2.id)) return  1;
        else                                                             return  0;
    }
}
Pattern B — Cleaner built-in compare (from CT-04 solution)
// DESCENDING by CGPA: swap the argument order for built-in compare
class CgpaComparator implements Comparator<Student> {
    public int compare(Student s1, Student s2) {
        return Double.compare(s2.cgpa, s1.cgpa);  // s2 first = descending
    }
}

// ASCENDING by ID: normal argument order
class IdComparator implements Comparator<Student> {
    public int compare(Student s1, Student s2) {
        return Integer.compare(s1.id, s2.id);      // s1 first = ascending
    }
}

★ Ascending vs Descending — The One Rule to Remember

  • Ascending (smallest first): Type.compare(s1.field, s2.field) — s1 first, normal order
  • Descending (largest first): Type.compare(s2.field, s1.field) — s2 first, swapped order

This swap trick works for all built-in types: Double.compare, Integer.compare, String.compareTo.

Using Comparators in Practice (from Main.java)

Anonymous Comparator (Inline — from Arraylist.pdf)

You can also write a comparator inline without creating a named class. This is useful for quick sorts in the same method.

// Sort by name ascending — anonymous comparator
Collections.sort(studentList, new Comparator<Student>() {
    public int compare(Student o1, Student o2) {
        return o1.name.compareTo(o2.name); // alphabetical ascending
    }
});

// Sort by id ascending — anonymous
Collections.sort(studentList, new Comparator<Student>() {
    public int compare(Student o1, Student o2) {
        return o1.id - o2.id; // works for int — safe subtraction
    }
});

// Sort by cgpa descending — anonymous (safe for double with if-else)
Collections.sort(studentList, new Comparator<Student>() {
    public int compare(Student o1, Student o2) {
        if (o2.cgpa > o1.cgpa) return  1; // o2 bigger → o2 first
                              return -1;
    }
});

⚠ Common Pitfalls — Comparator

  • Subtracting doubles: return (int)(o1.cgpa - o2.cgpa) is wrong — floating-point precision means 3.80 - 3.75 may round to 0, making two unequal students appear equal. Use Double.compare() or explicit if-else.
  • Forgetting to implement both methods: Comparator<T> only requires compare(), but students sometimes write compareTo() instead — which is the Comparable method. They have different names.
  • Getting ascending/descending backwards: The most common mistake. Always sanity-check: "what do I want to come FIRST?" If higher CGPA first, return -1 when o1.cgpa > o2.cgpa.
  • Passing the comparator to Arrays.sort() on a primitive array: Comparators only work on arrays of objects (Student[]), not primitive arrays (int[]). For primitives, use Arrays.sort(intArray) directly.
  • Using subtraction for ID when ID can be negative: If IDs are always positive, o1.id - o2.id is fine. But if IDs could be negative or large, integer overflow is possible. Safe habit: use Integer.compare(o1.id, o2.id).

ArrayList

ArrayList is a resizable array in Java. It is essential for this exam because the number of students in the file is not known in advance — you cannot use a fixed-size array. Every exam solution uses ArrayList to collect students before sorting and writing.

Core Concept — Why Not a Regular Array?

A regular array (Student[] arr = new Student[100]) requires knowing the size upfront. ArrayList grows automatically: when its internal capacity is exceeded, it creates a new array of double size, copies all elements, and continues. This makes it ideal for reading files of unknown length.

Complete Operation Reference

Declaring and using an ArrayList
import java.util.ArrayList;
import java.util.Collections;

ArrayList<String> list = new ArrayList<>(); // declare

// ADD
list.add("String 1");        // append to end — O(1) amortized
list.add(1, "String 4");   // insert at index 1 — shifts elements right

// GET
String s1 = list.get(0);     // get element at index 0
String s2 = list.get(2);     // get element at index 2

// FIND INDEX
int idx = list.indexOf("String 1"); // returns -1 if not found

// SEARCH
boolean has = list.contains("String 1"); // true/false

// MODIFY
list.set(1, "String 5");    // replace element at index 1

// REMOVE
list.remove(2);             // remove by index

// SIZE
int size = list.size();     // number of elements

// CLEAR
list.clear();               // removes all elements, size becomes 0

// ITERATE (for-each — most common in exam)
for (String s : list) {
    System.out.println(s);
}

HashSet (Brief — from Arraylist.pdf)

HashSet is introduced in the same lecture. Unlike ArrayList, it stores only unique elements — duplicates are silently ignored. Iteration order is not guaranteed.

HashSet<Integer> a = new HashSet<>();
a.add(33);  a.add(11);  a.add(33); // second 33 is ignored
System.out.println(a);           // prints [33, 11] or [11, 33] — order not guaranteed
a.remove(11);
System.out.println(a.size());    // 1
System.out.println(a.contains(33)); // true

⚠ Common Pitfalls — ArrayList

  • Calling Collections.sort(list) on a non-Comparable class: If Student does not implement Comparable, this throws a ClassCastException at runtime. Always provide a Comparator explicitly in this case.
  • Using raw type ArrayList without generics: ArrayList x = new ArrayList() (no <Type>) is legal but returns Object on every get() call, requiring a cast. Always specify the type.
  • remove(int index) vs remove(Object o): For ArrayList<Integer>, calling list.remove(1) removes the element at index 1, not the value 1. To remove by value, use list.remove(Integer.valueOf(1)). This is a classic exam trick.
  • IndexOutOfBoundsException: list.get(list.size()) is one past the end. Valid indices are 0 to size()-1.
  • Modifying the list inside a for-each loop: Calling list.remove() inside a for-each loop throws ConcurrentModificationException. Use an indexed for loop or collect indices to remove after.
  • contains() with the wrong type: In ArrayList x = new ArrayList() that contains an int 1, calling x.contains("1") returns false — the string "1" is not equal to the integer 1. The lecture code demonstrates this exact behaviour.

5 Rules You Must Never Break

★ The Non-Negotiables

  1. Always use split(",", -1) when detecting missing fields in CSV.
  2. Always call .close() on every Scanner and FileWriter.
  3. Never subtract doubles in a comparator — use Double.compare() or if-else.
  4. Always include return 0; at the end of every compareTo() / compare() method.
  5. Declare public static void main(String[] args) throws Exception to handle all File I/O exceptions cleanly.
