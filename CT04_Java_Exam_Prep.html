Section 01

Exam Paper Analysis (CT04)

The CT04 question is a single, integrated programming problem that requires all four syllabus topics working together. Understanding the structure before you write a single line of code is the highest-leverage exam skill you can develop.

What the Question Asks

You are given students.txt where each line is name,id,cgpa (comma-separated). Some lines may have one or more missing fields. Your program must produce three output files:

Output FileConditionSort Order
uga_candidate.txtCGPA ≥ 3.70CGPA descending (highest first)
probation.txtCGPA < 2.00ID ascending (most senior first)
errors.txtAny missing fieldNo sorting — preserve original order

Topic Weights & Exam Risk

TopicRole in the QuestionDifficultyMarks Risk
File I/O Read students.txt; write 3 output files; handle exceptions Medium ⚠️ High — forgetting .close() loses marks every time
ArrayList Three separate lists: uga, probation, errors Low ✅ Low risk — straightforward add & iterate
Comparator CgpaComparator (desc) + IdComparator (asc) — two separate classes Medium ⚠️ Medium — ascending/descending confusion is extremely common
Missing Value Detection Identifying incomplete lines → errors.txt High 🔴 Very High — most students miss the split(",", -1) requirement

Key Patterns to Recognise

  • Integration is deliberate. You cannot pass by knowing only one topic. Every part depends on the others.
  • The hidden trap: missing value detection. Students who use split(",") without -1 will silently drop trailing empty fields and miss some corrupt records entirely.
  • Two comparators, two sort directions. One ascending, one descending — mixing them costs full marks on both output files.
  • Three FileWriters, all must be closed. Forgetting any single fw.close() is a guaranteed runtime bug and a deduction on any marking scheme.
  • Language note: The question text says "Python program" — this is a template mismatch. The entire course, all slides, and the official solution use Java. Write in Java.
Section 02

Full Marks Strategy

Knowing the material is necessary but not sufficient. How you organise and present your answer matters just as much. Follow this blueprint on exam day.

Answer Structure — Always Write in This Order

Examiners mark top-to-bottom. This order maximises partial credit even if you leave something incomplete:

  1. Imports first, always. Write import java.io.*; and import java.util.*; as the very first lines. This prevents cascading compile errors and costs you nothing.
  2. Define the Student class. Fields, constructor, and toString(). This is low-risk, earns immediate marks, and everything else depends on it.
  3. Write all Comparator classes. Each is a self-contained class. Place them before main() so the compiler sees them when you call Collections.sort().
  4. Skeleton of main() — declare lists, open file, declare throws Exception. Put the structural bones in place before filling in logic. A skeleton with comments earns partial credit; empty space earns nothing.
  5. Fill the file-reading loop. Parse each line, check for missing values, create Student objects, add to the right list.
  6. Sort and write output files. One Collections.sort() + one FileWriter loop per output file. Close every FileWriter immediately after its loop.

Code Quality Tips That Score Marks

  • Use Double.compare(a, b) for doubles — never subtraction. Subtracting doubles produces a double, not an int, causing a compile error.
  • Descending = swap the arguments. Double.compare(s2.cgpa, s1.cgpa) (s2 first) gives descending order. This is the simplest, safest way.
  • Always call .trim() on parsed fields before calling isEmpty() or parseInt(). A space character is not empty, but it will crash parseInt().
  • Use split(",", -1) — the -1 argument tells Java to keep empty trailing fields. Without it, "Alice,001," returns only 2 elements, and your parts.length != 3 check still catches it — but "Alice,,3.85" returns 3 elements and an empty middle, which you also need to catch individually. Use -1 and check each field.
  • Close every file handle — open, write, close. If you forget even one fw.close(), data may not be flushed to disk.

Topic 01

ArrayList

An ArrayList is a dynamic, resizable array of object references. Unlike a fixed-size array, it can grow or shrink at runtime. When it runs out of space, Java automatically allocates a larger block of memory, copies all elements over, and frees the old memory — all transparently.

Declaration

import java.util.ArrayList;   // or import java.util.*;

ArrayList<String>  list     = new ArrayList<>();   // String list
ArrayList<Integer> nums     = new ArrayList<>();   // Integer list (NOT int)
ArrayList<Student> students = new ArrayList<>();   // Custom class list

Iterating

// Enhanced for-each (preferred — clean and safe)
for (Student s : students) {
    System.out.println(s.name);
}

// Index-based loop (use when you need the index)
for (int i = 0; i < students.size(); i++) {
    System.out.println(i + ": " + students.get(i).name);
}

Sorting an ArrayList

import java.util.Collections;

Collections.sort(students);                          // uses Comparable (compareTo)
Collections.sort(students, new CgpaComparator());  // uses Comparator (compare)
Collections.sort(list, Collections.reverseOrder()); // reverse natural order

HashSet — Brief Reference

A HashSet stores only unique elements and rejects duplicates silently. Order is not guaranteed.

HashSet<String> set = new HashSet<>();
set.add("Apple");
set.add("Apple");                       // duplicate — silently ignored
System.out.println(set.size());         // prints 1
System.out.println(set.contains("Apple"));  // true
set.remove("Apple");
System.out.println(set.contains("Apple"));  // false
⚠️ Common Pitfalls — ArrayList
  • Missing import: Forgetting import java.util.ArrayList; causes a compile error. Use import java.util.*; to cover everything at once.
  • Array syntax on ArrayList: Writing list[0] instead of list.get(0) is a compile error. ArrayList is an object, not an array.
  • remove(int) vs remove(Object): list.remove(2) removes the element at index 2. list.remove("Bob") removes the first occurrence of the String "Bob". These behave completely differently and the compiler won't warn you.
  • Primitive types: ArrayList<int> is a compile error. Use ArrayList<Integer>. Autoboxing handles the conversion automatically.
  • Hardcoding size: Never write i < 3 in a loop condition. Always use i < list.size().
  • Forgetting Collections.sort() needs import java.util.*: This is separate from the ArrayList import — they are both in java.util, so one wildcard import covers both.
✅ CT04 Application

You need three ArrayLists: ArrayList<Student> uga, ArrayList<Student> probation, and ArrayList<String> errors. Note that errors stores raw line strings, not Student objects. Declare all three before the file-reading loop.

Topic 02

Comparable

The Comparable<T> interface defines the natural (default) ordering of a class by embedding the comparison logic inside the class itself. Once implemented, Collections.sort(list) works without any extra argument — Java calls compareTo() automatically.

How to Implement

class Student implements Comparable<Student> {   // 1. Declare on class
    String name;
    String id;
    double cgpa;

    Student(String name, String id, double cgpa) {
        this.name = name; this.id = id; this.cgpa = cgpa;
    }

    public int compareTo(Student other) {           // 2. Override compareTo()
        // Negative → this comes BEFORE other  (ascending)
        // Positive → this comes AFTER  other
        // Zero     → equal, no swap
        if (this.cgpa > other.cgpa)      return -1;  // higher CGPA first (descending)
        else if (this.cgpa < other.cgpa) return  1;
        else {
            // Tie-break by ID ascending
            if      (Integer.parseInt(this.id) < Integer.parseInt(other.id)) return -1;
            else if (Integer.parseInt(this.id) > Integer.parseInt(other.id)) return  1;
            else return 0;
        }
    }
}

Calling It

Arrays.sort(array);           // for arrays  — calls compareTo()
Collections.sort(list);       // for ArrayList — calls compareTo()

Lecture Example Traced (Comparable.java)

Student s1 = new Student("Adrita", "001", 2.9);
Student s2 = new Student("Badhon", "002", 3.8);   // same CGPA as s3
Student s3 = new Student("Chamely", "003", 3.8);  // same CGPA as s2
Student[] array = {s1, s3, s2};
Arrays.sort(array);   // s2 and s3 tie on CGPA → tie-break by ID asc
Badhon: 002 cgpa:3.8
Chamely: 003 cgpa:3.8
Adrita: 001 cgpa:2.9

Badhon (ID 002) and Chamely (ID 003) both have CGPA 3.8, so the tie-break by ID ascending puts Badhon first. Adrita has the lowest CGPA and comes last.

The compareTo Return Value — How to Think About It

You returnMeaningEffect in sorted output
Any negative (e.g. -1)this ranks lower (comes earlier)this appears before other
0EqualRelative order unchanged
Any positive (e.g. +1)this ranks higher (comes later)this appears after other
⚠️ Common Pitfalls — Comparable
  • Missing the type parameter: Writing implements Comparable (raw) compiles but is a warning. Always write implements Comparable<Student> so compareTo takes a Student, not an Object.
  • Wrong return for descending order: Returning -1 when this.cgpa > other.cgpa gives descending (largest first). Many students instinctively return +1 here thinking "my CGPA is bigger so I'm 'bigger'" — this reverses the sort.
  • Double subtraction trick doesn't work: return (int)(this.cgpa - other.cgpa) fails for small differences (e.g. 3.80 − 3.75 = 0.05, cast to int = 0 → treated as equal). Always use explicit if-else.
  • Not handling the tie-break: If the spec says "sort by CGPA; ties broken by ID", returning 0 for equal CGPAs loses marks. You must implement the nested tie-break logic.
  • Arrays.sort() vs Collections.sort(): Arrays.sort() for arrays, Collections.sort() for ArrayList. Using the wrong one is a compile error.
Topic 03

Comparator

A Comparator<T> is a separate class that defines a custom ordering for objects without modifying their class. You can create as many Comparators as you need — one per sorting criterion — and pass whichever you need to Collections.sort().

🔵 Comparable
  • Logic lives inside the class
  • One natural order per class
  • Method: compareTo(T other)
  • Sort: Collections.sort(list)
  • Package: java.lang (auto-imported)
🟣 Comparator
  • Logic lives in a separate class
  • Unlimited orderings per class
  • Method: compare(T o1, T o2)
  • Sort: Collections.sort(list, comp)
  • Package: java.util (must import)

CGPA Comparator — Descending (from lecture CgBasedComparator)

class CgBasedComparator implements Comparator<Student> {
    public int compare(Student o1, Student o2) {
        if      (o1.cgpa > o2.cgpa) return -1;   // o1 first → highest CGPA at top
        else if (o1.cgpa < o2.cgpa) return  1;
        else                         return  0;
    }
}

ID Comparator — Descending (from lecture IDBasedComparator)

class IDBasedComparator implements Comparator<Student> {
    public int compare(Student o1, Student o2) {
        int id1 = Integer.parseInt(o1.id);
        int id2 = Integer.parseInt(o2.id);
        if      (id1 > id2) return -1;   // largest ID first
        else if (id1 < id2) return  1;
        else                return  0;
    }
}

ID Comparator — Ascending (from Main.java StudentIdComparator)

class StudentIdComparator implements Comparator<Student> {
    public int compare(Student s1, Student s2) {
        int left  = Integer.parseInt(s1.id);
        int right = Integer.parseInt(s2.id);
        if      (left > right) return  1;   // smallest ID first
        else if (left < right) return -1;
        else                  return  0;
    }
}

Cleaner Shorthand: Double.compare() and Integer.compare()

// Double.compare(a, b) returns negative if a < b, 0 if equal, positive if a > b

// Descending CGPA — swap s2, s1 to reverse the order
class CgpaDescComparator implements Comparator<Student> {
    public int compare(Student s1, Student s2) {
        return Double.compare(s2.cgpa, s1.cgpa);   // s2 first → descending
    }
}

// Ascending ID — natural order: s1 first
class IdAscComparator implements Comparator<Student> {
    public int compare(Student s1, Student s2) {
        return Integer.compare(s1.id, s2.id);      // s1 first → ascending
    }
}

Anonymous Inner Class (Inline Comparator — from lecture slides)

// No need for a separate named class — write the Comparator inline
Collections.sort(students, new Comparator<Student>() {
    public int compare(Student o1, Student o2) {
        return o1.name.compareTo(o2.name);  // sort by name ascending
    }
});

Ascending vs Descending — Quick Reference

GoalWhat compare(o1, o2) returnsShorthand
Ascending (small → large)negative when o1 < o2Double.compare(o1.x, o2.x)
Descending (large → small)negative when o1 > o2Double.compare(o2.x, o1.x) (swapped)
⚠️ Common Pitfalls — Comparator
  • Forgetting the import: Comparator is in java.util, not java.lang. Without import java.util.Comparator; (or import java.util.*;), implements Comparator won't compile.
  • Double subtraction is a compile error: return (int)(o1.cgpa - o2.cgpa) — the subtraction produces a double, and casting it to int silently discards precision (0.1 difference becomes 0, treated as equal). Use Double.compare() or explicit if-else.
  • Descending/ascending confusion: For descending, return -1 when o1 is BIGGER. This puts the bigger value at the front. Many students intuitively return +1 here, reversing the entire sort.
  • Passing the wrong Comparator: If you have CgpaComparator and IdComparator, double-check which one you pass to which Collections.sort() call. Swapping them gives completely wrong output and full marks lost.
  • Writing compareTo() instead of compare(): Comparator requires compare(T o1, T o2). If you write compareTo(), the interface is NOT satisfied — compile error or no sorting effect.
  • Integer.parseInt() on an int field: If id is declared as int in the Student class, do NOT call Integer.parseInt(s.id) — that's a method call on an int, which is a type error. Know whether your field is int or String.
✅ CT04 — Two Comparators Needed
  • CgpaComparator: return Double.compare(s2.cgpa, s1.cgpa) — descending, for uga_candidate.txt
  • IdComparator: return Integer.compare(s1.id, s2.id) — ascending, for probation.txt
  • In CT04, id is declared as int in Student (not String), so use Integer.compare() directly — no parseInt() needed.
Topic 04

File I/O

Java provides several classes for reading and writing text files. This course uses two patterns: Scanner + FileWriter (simple and direct) and BufferedReader + BufferedWriter (faster for large files). You must handle exceptions with either throws Exception on the method or a try-catch block.

Pattern 1: Scanner (Reading) + FileWriter (Writing)

This is the pattern used in the CT04 solution and in Main.java from lecture:

import java.io.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws Exception {

        // ── Reading ──────────────────────────────────────────
        Scanner sc = new Scanner(new File("students.txt"));  // FileNotFoundException
        while (sc.hasNextLine()) {
            String line = sc.nextLine();
            // parse line ...
        }
        sc.close();  // good practice

        // ── Writing ──────────────────────────────────────────
        FileWriter fw = new FileWriter("output.txt");   // IOException
        fw.write("Alice,001,3.85\n");
        fw.close();   // CRITICAL — data may not flush without this!
    }
}

Pattern 2: BufferedReader + BufferedWriter (from FileIO.pdf lecture)

import java.io.*;

public class Main {
    public static void main(String[] args) {
        try {
            BufferedReader reader = new BufferedReader(new FileReader("B.txt"));
            BufferedWriter writer = new BufferedWriter(new FileWriter("A.txt"));

            String s;
            while ((s = reader.readLine()) != null) {  // null = end of file
                writer.write(s);
                writer.newLine();   // platform-safe newline (\n on Linux, \r\n on Windows)
            }
            reader.close();
            writer.close();
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

Exception Handling — Two Valid Approaches

// Approach A — declare throws (cleaner for exam, fewer lines)
public static void main(String[] args) throws Exception { ... }

// Approach B — try-catch (explicitly handles the exception in-method)
try {
    // file operations
} catch (IOException e) {
    e.printStackTrace();
}

Either approach is acceptable. In exam conditions, throws Exception is faster to write and less error-prone.

Parsing CSV Lines + Missing Value Detection

This is the most important and most-failed pattern in CT04. Pay careful attention to the -1 argument in split:

String line = sc.nextLine();
String[] parts = line.split(",", -1);   // -1 preserves trailing empty fields!

// Without -1: "Alice,001," → parts = ["Alice","001"]  (length 2)
// With    -1: "Alice,001," → parts = ["Alice","001",""] (length 3, empty string at end)

if (parts.length != 3 ||
    parts[0].trim().isEmpty() ||
    parts[1].trim().isEmpty() ||
    parts[2].trim().isEmpty()) {
    errors.add(line);   // mark as corrupt
    continue;           // skip to next line
}

// Safe to parse — all three fields are present and non-empty
String name = parts[0].trim();
int    id   = Integer.parseInt(parts[1].trim());
double cgpa = Double.parseDouble(parts[2].trim());

Writing Multiple Output Files

// Standard pattern: open → write loop → close
FileWriter fw1 = new FileWriter("uga_candidate.txt");
for (Student s : uga)
    fw1.write(s.name + "," + s.id + "," + s.cgpa + "\n");
fw1.close();    // ← must not forget this

FileWriter fw2 = new FileWriter("probation.txt");
for (Student s : probation)
    fw2.write(s.name + "," + s.id + "," + s.cgpa + "\n");
fw2.close();    // ← must not forget this
⚠️ Common Pitfalls — File I/O
  • No throws Exception and no try-catch: File operations can throw checked exceptions. Without either, your code will not compile. This is an automatic and avoidable zero.
  • Forgetting fw.close(): FileWriter buffers writes in memory. If you don't close it, the buffer may never be flushed to disk. Your output files will be empty or truncated — a silent runtime bug. Close every FileWriter immediately after its write loop.
  • split(",") without -1 drops trailing empty fields: "Alice,001," split by "," gives ["Alice","001"] — only 2 elements. Your parts.length != 3 check catches this case anyway, but split(",", -1) is the correct and explicit approach that makes your intent clear. Always use it when detecting missing values.
  • Not trimming: If the input file has spaces around commas ("Alice , 001 , 3.85"), then parts[1] is " 001 " and Integer.parseInt(" 001 ") throws a NumberFormatException. Always call .trim() before parsing.
  • sc.next() vs sc.nextLine(): sc.next() reads one space-delimited token. sc.nextLine() reads the entire line including commas. For CSV parsing, always use nextLine().
  • reader.readline() (lowercase l): The correct spelling is readLine() with a capital L. The lowercase version does not exist — compile error.
  • sc.hasNext() vs sc.hasNextLine(): When reading line by line, use hasNextLine(). hasNext() checks for the next token, not the next line.

📄 CT04 Problem Statement (Restated)

File students.txt contains comma-separated student records (name, id, cgpa). Some lines have missing values. Write a Java program that produces:

  • uga_candidate.txt — students with CGPA ≥ 3.70, sorted by CGPA descending
  • probation.txt — students with CGPA < 2.00, sorted by ID ascending
  • errors.txt — all lines with one or more missing values

Step 1 — Imports and Student Class

ArrayList File I/O

Declare both wildcard imports at the top. The Student class uses int id (not String) so Integer.compare() works directly without parseInt().

import java.io.*;
import java.util.*;

class Student {
    String name;
    int id;        // int, not String — simplifies comparator
    double cgpa;

    Student(String name, int id, double cgpa) {
        this.name = name;
        this.id   = id;
        this.cgpa = cgpa;
    }

    public String toString() {
        return name + ", " + id + ", " + cgpa;
    }
}

Step 2 — Two Comparator Classes

Comparator

Write these before main(). Each class handles exactly one output file's sort order.

// uga_candidate.txt needs CGPA descending: swap s2, s1 in Double.compare
class CgpaComparator implements Comparator<Student> {
    public int compare(Student s1, Student s2) {
        return Double.compare(s2.cgpa, s1.cgpa);  // s2 first = descending
    }
}

// probation.txt needs ID ascending: s1 first in Integer.compare
class IdComparator implements Comparator<Student> {
    public int compare(Student s1, Student s2) {
        return Integer.compare(s1.id, s2.id);     // s1 first = ascending
    }
}

ArrayList File I/O

    public static void main(String[] args) throws Exception {

        ArrayList<Student> uga       = new ArrayList<>();
        ArrayList<Student> probation = new ArrayList<>();
        ArrayList<String>  errors    = new ArrayList<>();  // stores raw lines

Step 4 — Reading Loop: Parse, Validate, Categorise

File I/O ArrayList

The -1 in split is the most commonly missed detail in this entire question. Without it, a trailing empty field is silently dropped and the error isn't detected correctly.

        while (sc.hasNextLine()) {
            String line = sc.nextLine();
            String[] parts = line.split(",", -1);  // -1 keeps trailing empty strings

            // Detect missing values: wrong count, OR any field is blank after trim
            if (parts.length != 3 ||
                parts[0].trim().isEmpty() ||
                parts[1].trim().isEmpty() ||
                parts[2].trim().isEmpty()) {
                errors.add(line);   // add the raw corrupt line
                continue;           // skip — don't try to parse it
            }

            // Parse — safe because we validated above
            String name = parts[0].trim();
            int    id   = Integer.parseInt(parts[1].trim());
            double cgpa = Double.parseDouble(parts[2].trim());
            Student s = new Student(name, id, cgpa);

            // Categorise into the right list
            if      (cgpa >= 3.70) uga.add(s);
            else if (cgpa < 2.00) probation.add(s);
            // Students with 2.00 <= cgpa < 3.70 are not in any output list
        }
        sc.close();

Step 5 — Sort and Write Output Files

Comparator File I/O

One sort + one write loop per output file. Note: errors.txt requires no sorting — just write in the order received.

        // ── uga_candidate.txt: CGPA descending ──────────────────
        Collections.sort(uga, new CgpaComparator());
        FileWriter fw1 = new FileWriter("uga_candidate.txt");
        for (Student s : uga)
            fw1.write(s + "\n");    // calls s.toString() automatically
        fw1.close();

        // ── probation.txt: ID ascending ─────────────────────────
        Collections.sort(probation, new IdComparator());
        FileWriter fw2 = new FileWriter("probation.txt");
        for (Student s : probation)
            fw2.write(s + "\n");
        fw2.close();

        // ── errors.txt: no sort needed ──────────────────────────
        FileWriter fw3 = new FileWriter("errors.txt");
        for (String e : errors)
            fw3.write(e + "\n");
        fw3.close();
    }
}

Complete Program — Copy-Ready

import java.io.*;
import java.util.*;

class Student {
    String name; int id; double cgpa;
    Student(String name, int id, double cgpa) {
        this.name = name; this.id = id; this.cgpa = cgpa;
    }
    public String toString() { return name + ", " + id + ", " + cgpa; }
}

class CgpaComparator implements Comparator<Student> {
    public int compare(Student s1, Student s2) {
        return Double.compare(s2.cgpa, s1.cgpa);  // descending
    }
}

class IdComparator implements Comparator<Student> {
    public int compare(Student s1, Student s2) {
        return Integer.compare(s1.id, s2.id);     // ascending
    }
}

public class Main {
    public static void main(String[] args) throws Exception {
        ArrayList<Student> uga       = new ArrayList<>();
        ArrayList<Student> probation = new ArrayList<>();
        ArrayList<String>  errors    = new ArrayList<>();

        Scanner sc = new Scanner(new File("students.txt"));
        while (sc.hasNextLine()) {
            String line = sc.nextLine();
            String[] parts = line.split(",", -1);
            if (parts.length != 3 || parts[0].trim().isEmpty() ||
                parts[1].trim().isEmpty() || parts[2].trim().isEmpty()) {
                errors.add(line); continue;
            }
            String name = parts[0].trim();
            int    id   = Integer.parseInt(parts[1].trim());
            double cgpa = Double.parseDouble(parts[2].trim());
            Student s = new Student(name, id, cgpa);
            if      (cgpa >= 3.70) uga.add(s);
            else if (cgpa < 2.00) probation.add(s);
        }
        sc.close();

        Collections.sort(uga, new CgpaComparator());
        FileWriter fw1 = new FileWriter("uga_candidate.txt");
        for (Student s : uga) fw1.write(s + "\n");
        fw1.close();

        Collections.sort(probation, new IdComparator());
        FileWriter fw2 = new FileWriter("probation.txt");
        for (Student s : probation) fw2.write(s + "\n");
        fw2.close();

        FileWriter fw3 = new FileWriter("errors.txt");
        for (String e : errors) fw3.write(e + "\n");
        fw3.close();
    }
}

String name o1.name.compareTo(o2.name) o2.name.compareTo(o1.name)
String id (stored as String) Integer.compare(Integer.parseInt(o1.id), Integer.parseInt(o2.id)) Swap o2 and o1 in the above

Pre-Exam Self-Check

  • Added import java.io.*; and import java.util.*; at the very top?
  • Declared throws Exception on main() (or wrapped everything in try-catch)?
  • Used split(",", -1) with the -1 argument?
  • Called .trim() on each part before isEmpty(), parseInt(), or parseDouble()?
  • Closed every FileWriter with fw.close() after its write loop?
  • Passed the correct Comparator to each Collections.sort() call?
  • For CGPA descending: Double.compare(s2.cgpa, s1.cgpa) — s2 comes first?
  • For ID ascending: Integer.compare(s1.id, s2.id) — s1 comes first?
  • If id is a String field: used Integer.parseInt() inside the comparator?
  • compareTo() for Comparable; compare() for Comparator — not mixed up?
